# Product Proposal: TaskPilot Agent

### Обоснование идеи
**Прикладная задача** — освободить руководителей и специалистов от ручного ведения бэклога и отслеживания зависимостей. Агент выступает автономным ассистентом, который превращает хаотичные сообщения (чат, почта, голос) в структурированные задачи, автоматически фиксируя проблемы и связи между ними.
**Глубинная проблема** — задачи теряются в переписке, зависимости забываются, а история проблем не накапливается. Существующие трекеры требуют ручного ввода, что снижает актуальность данных.

### Цель и метрики
**Цель:** PoC-система, которая через чат-интерфейс позволяет создавать, обновлять задачи и отслеживать зависимости с использованием семантического поиска, обеспечивая строгую изоляцию доступа пользователей.

**Метрики успеха:**
*   **Продуктовые:** Точность связывания сообщений с существующими задачами (>85% через семантический поиск), доля задач с автоматически заполненным полем "проблема" (>50%).
*   **Агентские:** Точность извлечения сущностей (дедлайн, статус, зависимость) по сравнению с ручным разметчиком (>90%), способность корректно запросить уточнение при неоднозначности (100% случаев сомнений).
*   **Технические:** Время ответа на запрос (p95) <5 сек (без учета LLM latency), строгость изоляции данных (0 утечек между пользователями), доступность PostgreSQL и Vector DB >99%.

### Сценарии использования (включая edge‑кейсы)
*   **Основной:** Пользователь пишет в чат: "Нужно подготовить отчет до пятницы, жду данные от Иванова". Агент создает задачу, ставит дедлайн, фиксирует проблему "ожидание данных".
*   **Обновление:** Пользователь: "По отчету данные пришли". Агент находит задачу семантически, обновляет статус проблемы, меняет статус задачи на "В работе".
*   **Зависимости:** Пользователь: "Задача Б не начнется пока не закроется Задача А". Агент создает связь в журнале зависимостей.
*   **Саммари:** Пользователь: "Покажи статус по отчету и что его стопорит". Агент формирует сводку из карточки задачи и журнала зависимостей.
*   **Edge‑кейсы:**
    *   **Неоднозначная ссылка:** "Обнови ту задачу по маркетингу" (их несколько) → Агент спрашивает: "Вы имеете в виду задачу X или Y?".
    *   **Нет доступа:** Пользователь запрашивает задачу другой группы → Агент отвечает: "У вас нет доступа к этой задаче".
    *   **Противоречие:** В сообщении статус "Готово", но есть активные зависимости → Агент предупреждает о конфликте.

### Ограничения
*   **Технические:** p95 latency <10 сек (при поиске по векторам и LLM-анализе), актуальность векторного индекса (задержка синхронизации PG ↔ FAISS < 1 мин).
*   **Операционные:** Строгое соблюдение прав доступа (RLS на уровне БД + проверка в коде), обработка только русского языка (MVP).
*   **Данные:** Векторная база хранит только эмбеддинги текстов задач, персональные данные в явном виде не векторизуются.

### Архитектурный набросок
**Модули:**
1.  **Input Adapter:** Нормализация входа (чат/почта/голос → текст + user_id).
2.  **Semantic Search Engine:** Векторизация запроса, поиск похожих задач в FAISS.
3.  **Task Manager:** CRUD операций над карточками задач в PostgreSQL.
4.  **Dependency Logger:** Ведение журнала связей (статус, дата, ID задач).
5.  **Agent Core:** Анализ намерений, извлечение сущностей, генерация ответов.
6.  **Access Control:** Проверка прав (my/assigned/shared) перед любым действием.

**Интеграции:** PostgreSQL (основное хранение), FAISS (семантический поиск), LLM API (анализ и генерация).
**LLM:** Для извлечения сущностей, семантического сравнения, генерации саммари, формулировки уточняющих вопросов.

**Data Flow:**
```text
1. Запрос пользователя → Input Adapter (текст + контекст).
2. Agent Core выявляет топик → Semantic Search Engine ищет кандидаты в FAISS.
3. Agent определяет ID задач → Access Control проверяет права.
4. Task Manager / Dependency Logger обновляют PostgreSQL.
5. Ответ пользователю (структурированный + ID задач).
